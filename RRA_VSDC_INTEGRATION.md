# RRA VSDC API Integration & CIS Compliance Guide

This document provides a comprehensive blueprint to integrate the **Rwanda Revenue Authority (RRA) Virtual Sales Data Controller (VSDC)** and achieve **Certified Invoicing System (CIS)** compliance in the **Big Company Backend** project.

---

## 1. Architectural Concept & Core Rules

The VSDC acts as a local bridge system between the **Big Company Backend** (CIS/ERP) and the **RRA EBM 2.1 API Server**.

```
┌────────────────────────┐         ┌──────────────┐         ┌───────────────┐
│  Big Company Backend   │  HTTP   │  Local VSDC  │  HTTPS  │ RRA EBM Server│
│ (Express / TypeScript) ├────────>│ (Tomcat/Jetty)├────────>│     (RRA)     │
└────────────────────────┘  JSON   └──────────────┘  JSON   └───────────────┘
```

### Core Compliance Rules
1. **No-Stand-Alone Rule**: The CIS cannot produce invoices or receive importation/purchase data unless connected to a functional VSDC unit assigned to the same TIN.
2. **Offline Mode**: If internet connection is lost, VSDC can sign and buffer receipts locally for up to **24 hours**. After 24 hours without connectivity, the VSDC will stop issuing receipt numbers and signing transactions.
3. **Synchronous Validation**: The CIS must wait for a VSDC response before finalizing a transaction or printing a receipt.
4. **Resiliency**: The CIS must implement a **1000 ms timeout** for receiving responses from VSDC. If it fails, retry; after a configured number of failures, display a warning about VSDC disconnection.
5. **No Double Corrections**: A transaction cannot be edited directly; it must be canceled first (using transaction refund/reversal referencing the original VSDC receipt number). A transaction can only be canceled once.

## 2. Receipt Labels & Transaction Types

Every receipt printed or generated by the CIS must fall under one of these specific combinations:

| Receipt Type | Transaction Type | Receipt Label | Signature Required? |
| :--- | :--- | :--- | :--- |
| **NORMAL** | SALE | **NS** | Yes |
| **NORMAL** | REFUND | **NR** | Yes |
| **COPY** | SALE | **CS** | Yes |
| **COPY** | REFUND | **CR** | Yes |
| **TRAINING** | SALE | **TS** | No |
| **TRAINING** | REFUND | **TR** | No |
| **PROFORMA** | SALE | **PS** | No |

> [!WARNING]
> For **COPY (CS/CR)**, **TRAINING (TS/TR)**, and **PROFORMA (PS)** tickets, they must be clearly distinguishable. You must print a watermark or header displaying their type, and place the following warning text below the totals in double size: **"THIS IS NOT AN OFFICIAL RECEIPT"**.

## 3. Database Schema Additions (`prisma/schema.prisma`)

To support compliance, extend the database schema:
* **`Sale` / `Invoice` Table**:
  - `rraInternalData` (String?): Cryptographic internal representation.
  - `rraReceiptSignature` (String?): Signature token from VSDC.
  - `rraReceiptNumber` (Int?): The VSDC-issued sequential number.
  - `rraResultCd` (String?): Response code from the local service.
  - `receiptLabel` (String): e.g., "NS", "NR", "CS", etc.
  - `mrc` (String): Machine Registration Code.
* **`Product` / `Item` Table**:
  - `itemClsCd` (String): 10-digit RRA item classification code.
  - `itemTyCd` (String): Item type (1=raw material, 2=finished product, etc.).
  - `taxTyCd` (String): Tax rate group (`A`=Exempt, `B`=18%, `C`=Exempt/Zero-rated, `D`=Non-taxable).
  - `qtyUnitCd` (String): Quantity unit (e.g., `U`, `KG`, `LTR`).
  - `pkgUnitCd` (String): Packaging unit (e.g., `NT`, `BE`).

## 4. Main Endpoints & Payload Contracts

Depending on the VSDC version, communication uses JSON (v1.0.5) or legacy XML (v1.0).

### A. EBM 2.1 JSON APIs (Newer VSDC)

#### 1. Device Initialization (`POST /initializer/selectInitInfo`)
```json
{
  "tin": "999993204",
  "bhfId": "00",
  "dvcSrlNo": "dvc999993204"
}
```
* **Response**: Returns success code (`000`), SDC ID, and cryptographic keys.

#### 2. Sales Transaction (`POST /trnsSales/saveSales`)
```json
{
  "tin": "999993204",
  "bhfId": "00",
  "invcNo": 125,
  "orgInvcNo": 0,
  "custTin": "100600570",
  "custNm": "Customer Name",
  "salesTyCd": "N",
  "rcptTyCd": "S",
  "pmtCd": "01",
  "totAmt": 11800.00,
  "itemList": [
    {
      "itemSeq": 1,
      "itemCd": "RW1NTXU0000006",
      "itemClsCd": "5059690800",
      "itemNm": "Piped Gas",
      "qty": 1.00,
      "prc": 11800.00,
      "totAmt": 11800.00,
      "taxTyCd": "B",
      "taxblAmt": 10000.00,
      "taxAmt": 1800.00
    }
  ]
}
```
* **Response**: Returns `rcptNo`, `intrlData`, and `rcptSign`.

---

### B. Legacy XML Protocol Command Contracts (v1.0)
If integrating with older VSDC clients, format inputs as XML nodes under `<TIN>`, `<CMD>`, and `<DATA>`.

* **Send Receipt (`CMD: SEND_RECEIPT`)**:
  - **Payload Format**: CSV representation of receipt parameters inside the `<DATA>` block:
    ```xml
    <TIN>999993204</TIN>
    <CMD>SEND_RECEIPT</CMD>
    <DATA>NS,S,MRC12345,999993204,24/06/2026 17:40:00,125,0,18.00,0,0,0,0,10000.00,0,0,0,1800.00,0,0,0,100600570</DATA>
    ```
* **Read Signature (`CMD: RECV_RECEIPT`)**:
  - Reads back signatures and SDC counts.
  - **Response**: CSV format: `VSDC_SERIAL,RCT_TYPE_COUNT,TOTAL_COUNT,LABEL,DATE_TIME,SIGNATURE,INTERNAL_DATA`.

## 5. Required Receipt Layout & Print Specifications

Every receipt printed by the CIS must display a designated **"SDC INFORMATION"** footer containing:

```
--------------------------------------------------
SDC INFORMATION
Date: DD/MM/YYYY Time: HH:MM:SS (Stamps from VSDC)
SDC ID: SDCXXXXXXXXX
RECEIPT NUMBER: [Type_Counter]/[Total_Counter] [Label] (e.g., 168/258 NS)
Internal Data: XXXX-XXXX-XXXX-XXXX-XXXX (Hyphenated every 4 characters)
Receipt Signature: XXXX-XXXX-XXXX-XXXX (Hyphenated every 4 characters)
--------------------------------------------------
[QR CODE]
--------------------------------------------------
RECEIPT NUMBER: [CIS_Receipt_Number]
DATE: DD/MM/YYYY TIME: HH:MM:SS (Stamps from CIS)
MRC: BBBCCNNNNNN
```

* **QR Code Payload**: Constructed as:
  `invoice_date(ddmmyyyy)#time(hhmmss)#sdc_number#sdc_receipt_number#internal_data#receipt_signature`

## 6. Daily Audits & Reporting (X, Z, and PLU Reports)

The CIS must support three reporting structures:

1. **X Daily Report**:
   * Summary of operations since the last Z report to the present.
   * Includes: Total NS Sales, Total NR Refunds, taxable amounts per rate, deposits, void counters, payment method breakdowns, and incomplete sales counts.
2. **Z Daily Report**:
   * The formal daily closing accounting record (runs from 00:00:00 to 23:59:59).
   * Summarizes the same financial metrics as the X report but resets daily totals.
3. **PLU (Price Look-Up) Report**:
   * Detail report showing quantities sold, amounts collected per item/category, and remaining stock quantities within a user-chosen date interval.

## 7. Step-by-Step Implementation Roadmap

1. **Prisma Setup**: Push changes to MySQL schema using `npx prisma db push`.
2. **VSDC Service**: Create `src/services/vsdc.service.ts` to handle formatting (JSON or XML depending on your active VSDC specification), timeouts, and offline queues.
3. **Checkout Integration**: Inject verification logic in sales controllers. Roll back DB actions if VSDC rejects the invoice.
4. **Auditing**: Create controllers to compile transactional data into daily X/Z reports and PLU lists.
